## 0. 一句话结论(已翻转)

最初设想把生成从固定 DAG 升级成「目标驱动的有界质量闭环」(评估→不达标→自动返工)。
但**上生产实测两个世界(甄嬛传/凡人修仙传),质量都很高**(详见 §3)——06-22 的修复已见效,must_have 全齐、IP 一致性强。
**闭环的全部价值来自「经常返工」,质量稳定时它就是纯增时间+复杂度的过度工程。**
所以现阶段:**① critic 瘦身**(把不闭环就没意义的 LLM 软评分从同步主流程摘掉);
**② 异步打分**(生成完后台跑一次评分,不阻塞);**③ admin 观察**(攒真实失败率数据)。
**闭环转 backlog**:等观察到一批低分,再有据地做(触发条件见 §9)。

## 1. 什么是真正的 Agent(思考记录,本 plan 地基)

Agent 的本质是**自主性**,但分两种,对生成任务必须分清:

| | 对「过程」的自主 | 对「结果」的自主 |
|---|---|---|
| 含义 | 自己决定先做哪步、用什么工具、什么顺序 | 自己对照目标判断差距、决定补哪、决定何时算达标 |
| 形态 | 自由 ReAct / planner agent | 有界质量闭环 |
| 对生成世界 | **负债**:骨架本就确定(世界观→角色→事件),自由决定顺序毫无收益,且毁掉可观测/SSE进度/成本可控/阶段并发 | **才是缺的**:critic 抓到主角丢失只 warning、草稿照存 |

结论:真正的 Agent ≠ 自由流程。对结构化产物,「对结果负责」才是 agentic 核心。
**但「对结果负责」不一定要做实时闭环——也可以是「事后异步打分 + 人工兜底」**,在质量已稳时这才是对的成本结构。

## 2. 现状诊断(已读代码核实)

### 2.1 12 段固定 DAG

```
research → ip_research → world_base → C1/C2并发 → D1/D2并发 → roster
→ shared_events → relations → events_data → playable
→ critic → visual_brief → images → validating
```

### 2.2 critic 段现在干 5 件事(`_run_critic` + `world_critic_service.py`)

| 子项 | 是什么 | 不闭环时的价值 |
|---|---|---|
| H1 形状校验(Python,0token) | 缺字段/空数组挡住 | **真在挡崩坏** |
| must_have backfill(确定性) | 主角丢了用 ip_pack 补回 | **真在补人**(06-22 成果) |
| H3 moderation | 内容合规 | 法务底线 |
| H2 轻 critic(LLM) | lore/事件 IP 一致性打分 | **只产 warning,没人看 = 纯烧钱** |
| H2.5 重 critic(LLM) | 角色/可玩一致性,会就地微调 | 微调一点点用,ROI 低 |

**致命点**:H2/H2.5 产出 `quality_warnings` 文本,除 must_have backfill 外**不回头重做任何阶段**,世界照样发布。

### 2.3 关键基线数据(本地 96 个真实生成任务 + 生产实测)

- 文本 LLM 成本:p50 **$0.21** / 均 $0.44 / 26.7K token。
- 图片:**$0.20/张**,大 IP 8-12 张 ≈ $1.6-2.4 → **图片占单次总成本 ~85%**。
- 世界生成时长:均 **628 秒 ≈ 10.5 分钟**(44 个世界任务)。

**结论:质量已够好,闭环现阶段没必要。**

## 4. 落地方案

### 4.1 critic 瘦身

主同步流程(critic 段)**只留硬的**:
- ✅ 保留 H1 形状校验(Python)、**must_have backfill(安全网,必须留)**、H3 moderation(合规)。⚠️ 去掉的只是「**只打分不动手的 H2/H2.5**」；backfill 是确定性地**真把丢的主角补回来**(06-22 的疗效靠它),去掉它质量会退回 06-22。
- ➡️ 把 **H2 轻 critic + H2.5 重 critic(LLM 软评分)摘出同步流程**,改为 §4.2 的**按需**软评分。

收益:主流程少 2 个 LLM 调用 + 重 critic →**真收益是「提速 + 简化」**,不是省钱(图片占单次 ~85% 成本,文本侧这点是零头)。质量不掉(软分本来就只 warning、不返工)。

### 4.2 异步打分

```
世界生成 SSE done(照常发布草稿,不等打分)
        │ done 之后 fire-and-forget(失败不影响生成)
        └─→ QualityScore 协程(★每个世界都自动跑——这才攒得到数据):
              ├─ Python 硬指标($0):角色数/可玩数/must_have覆盖/结构完整/events·shared_events 条数
              ├─ LLM 软评分(IP一致性/撞车/张力,原 H2/H2.5):**默认就跑**,产分 + 一句理由
              │     成本 ~$0.0x/世界(文本侧,远小于图片 $1.6-2.4),为攒数据值得
              ├─ 安全网触发量(额外盲点信号):backfill_invoked/must_have_rescued/roster_pruned/soft_warnings 数
              └─ 存 world_quality_scores 表(task_id + 硬指标 + 软分及理由 + 安全网触发量 + 打分时刻)
```

> **软分负责"质量高不高",安全网触发量负责"这分是不是靠补救撑的"——两个一起看。**
> 为什么要单独记触发量:must_have backfill 在 done **之前**就把丢的主角补齐了 → "最终 must_have 齐不齐"**永远满分** → 光看它会误判"都好"。但 backfill 只管主角在不在,管不了内容好不好,所以**质量本身仍要靠软分**(这就是软分默认全开的原因);触发量只是防"软分/完整度看着高、其实是补救撑着"。

### 4.3 admin 观察(详见 §6)

generations 详情页看单次质量卡 + 列表加质量列筛低分。

## 5. 钉死的 8 个设计决策(防做歪)

1. **趋势对比只用 Python 硬指标。** LLM 软分换模型/prompt 就变、不可纵向比;软分只作单次参考,不进趋势,不主导决策。
2. **打分锁定「生成完成快照」,不追草稿后续编辑。** 分数关联 `task_id`、锁死在 done 那一刻——我们观察的是引擎吐出的质量,不是人工修过的。
3. **真异步,不阻塞。** done 照常发,打分 fire-and-forget,打分失败不影响生成。净成本增量≈0(软分只是从同步挪到异步)。
4. **不能是死看板。** 低分必须能一键跳世界详情/草稿、能触发重生成/下架。
5. **critic 瘦身留安全网。** H1/must_have backfill/moderation 必须留同步主流程,别瘦过头。
6. **反范围蔓延。** MVP 只做:详情页一张质量卡 + 列表一列分数。雷达图/仪表盘趋势是 P2。
7. **软分默认全开,质量本身靠它。** 硬指标 + LLM 软分都每个世界自动打——目的就是攒质量数据。软分费钱但 ~$0.0x/世界、远小于图片,值得默认开。
8. **额外记「安全网触发量」,作盲点补充(非替代打分)。** must_have backfill 在 done 前补齐主角 → "最终完整度"永远满分 → 单看它会漏判。故另记 backfill/prune 触发量:软分告诉你"质量高不高",触发量告诉你"是不是靠补救撑的"。⚠️ 触发少 ≠ 质量好(只代表主角没丢)。
   > (原决策⑨「软分不进纵向趋势」已并入决策①,不再单列。)

## 6. admin:在哪看、看什么

### 6.1 在哪看 —— 主场 `generations`,不是 `content`

| | generations | content |
|---|---|---|
| 覆盖 | **所有生成,含 private 草稿** | 只有已发布/已下架 |
| 语义 | 生成结果质量,正对「观察引擎」 | 治理已上线内容 |
| 已有承载 | 详情页已展示事件流+warning,质量卡自然延伸 | 动作是下架,偏 |

→ 主入口 `generations`(覆盖草稿是关键,质量问题大多在发布前就该看到)。`content` 页**可选**加质量列辅助下架决策。

### 6.2 看什么 —— MVP 只两处

1. **`generations/[taskId]` 详情页 — 加「质量报告」卡**(挂事件流上方):
   - **硬指标表**(带达标线+红绿):must_have `4/4`✓、角色数 `18`✓、可玩 `12`✓、结构完整 `95%`✓
   - **软指标(默认就有,质量主力)**:IP一致性 8/10「纯元梗都准」、撞车 9/10、张力 7/10
   - **安全网触发(配合软分一起看)**:backfill `未触发`✓ / `补了2个主角`⚠️、prune 删除 `0`、soft_warnings `1`
   - 顶部总评 badge + 「查看世界」「重新生成」入口(对应决策④)
2. **`generations` 列表 — 加「质量」列**:分数 badge,低分红色;可按质量排序、筛低分。**这列就是观察失败率的主力——扫一眼有没有一批红的。**

**P2(攒够数据再做):** 仪表盘加「近 30 次 硬指标趋势」小图。

## 9. backlog:闭环什么时候才做

**触发条件**(满足任一再考虑实时闭环,届时重启本 plan 的闭环设计):
- **安全网触发率持续偏高**(如 >20% 的生成需要 backfill 补主角 / prune 大量删角色)——注意**不能用「done 时 must_have 不全」**(backfill 兜底后它永远是 0,看不出问题);要看的是**网被迫介入的频率**。
- 角色数 / 可玩数等**最终态硬指标**仍有失败(backfill 补不了的那类,如整体 roster 偏小)。
- 出现**某类系统性低分**(如某 genre / fidelity_mode 稳定差),且 prompt/research 层(06-22 路线)已优化到头仍不行。

**闭环设计要点(备查,届时复用):** 评审官产出结构化 Verdict(rubric 硬分 + LLM 软分 + 定位到 target_stage 的 fix 指令)→ 修复编排器带 fix 有界重做对应段(放 images **之前**,零图片返工)→ 预算守卫(max 2 轮 + 收敛性检测,同一 failure 无改善则提前放弃)。成本增量约 +2-4%(图片零返工),时间增量 +10-15%(串行重做)。

# 模块文档模板

> 这是 `docs/modules/<name>.md` 的统一模板。**每篇模块文档以"能力矩阵"为核心**——按子领域把模块的能力点全部铺开，标状态，让读者一眼看到：模块现在能做什么、有哪些 gap、哪些可以优化。
>
> 模板参考实现：[`npc.md`](./npc.md) — 它是被 review 过的样板（§1 能力矩阵 8 组、30+ 能力点）。所有新模块文档按这个密度写。

---

## 文档元信息（必写）

第一段必须是引用块状态声明：

```markdown
> 状态截至 YYYY-MM-DD。覆盖 ... 落地后的形态。
```

模块文档**会过期**，写明"截至某日"逼读者去 grep 验证。

## 1. 模块是干什么的（必写，3-5 句）

回答：

- 这个模块在系统里担任什么角色
- 边界：相邻职责不在这里做，去看哪个模块
- 紧密耦合的上下游

避免重复 ARCHITECTURE.md 的全局介绍——这里只讲**这个模块**。

## 2. 能力矩阵（必写，核心，文档主菜）

**这是模块文档最重要的部分**。按子领域分组，把所有能力点铺开。读者扫一遍就知道：

- 这个模块能做什么
- 哪些是真做了 / 哪些挂着没做 / 哪些走了妥协路径
- 哪些是可优化的 gap

### 状态符号统一

| 符号 | 含义 |
|---|---|
| ✅ | 已落地，跑得动 |
| 🟡 | 部分落地 / 走的妥协路径（注明妥协点） |
| 🔵 | 已设计但还没动手（spec 在 / 计划在） |
| ❌ | 未实现，待评估 |

### 矩阵密度要求

- **每个子领域至少 4-8 个能力点**（不是 1-2 个）
- 子领域至少 3-5 组
- 总能力点 **≥ 20 个**——少于这个数说明你想得不够细
- 每个能力点：一句话名字 + 状态 + 实现位置（文件名 / 函数名 / 表名）

### 子领域分组示例

每个模块自己定，但保持"维度清晰"。常见维度：

- **核心能力**（最主要做的事）
- **数据维度**（涉及哪些数据 / 状态）
- **协作维度**（跟别的模块怎么配合）
- **可观测 / 性能**
- **安全 / 隔离 / 边界**（如适用）
- **配置 / 运维**

参考 npc.md 用 A-H 八组：角色认知 / 情境感知 / 历史认知 / 关系认知 / 行为约束 / 信息隔离 / 性能可观测 / Director 协作。

### 表格形式

```markdown
### A. <子领域名>（"<一句话方括号副标题>"）

| 能力 | 状态 | 实现 |
|---|---|---|
| 名字 | ✅ | npc_name 字段 |
| 性格 | ✅ | personality |
| 角色身份/职业/年龄/外貌 | ❌ | schema 没专门字段，未来扩展 |
```

## 3. 关键能力实现要点（推荐）

不是所有能力都要展开——挑 **3-6 个最值得讲清楚的**，每个写几句话讲实现机理 + 关键文件 + 设计取舍。

每节按这个结构：

```markdown
### 3.X 能力名（对应矩阵某行）

**问题**：之前的状态/痛点
**解决**：现在怎么做（一两句）
**实现**：
- 关键文件 / 函数
- 关键算法 / 数据结构
- 配置开关（如果有）
**取舍**：为什么不是另一种做法（可选，但有非显然取舍时必写）
```

避免堆代码片段——指出位置即可。复制超过 5 行代码到文档**保证**会失同步。

## 4. 信息隔离 / 边界（按需）

只在涉及"什么数据该被谁看见"的模块写（NPC / memory / auth）。三栏：

- ✅ 应该看到的（正向清单）
- ❌ 不应该看到的（负向清单 + 原因 + 当前怎么排除）
- 🟡 灰色区域 / 设计决策

这是模块的**安全契约**。新加 prompt 段或新拉数据时必须对照。

## 5. system prompt 完整结构（仅 LLM agent 模块）

只在涉及构造 LLM system prompt 的模块写（NPC / Director / Narrator）。把结构以 fenced code block 完整列出：

```
[稳定前缀]
...

[可变后缀]
...
```

理由：cache-friendly prompt 结构（稳定前缀 + 可变后缀）是 1.A.3 的关键约束，prompt 改动必须按这个分层。

## 6. 关键代码位置速查（必写）

表格列出本模块涉及的文件 + 一句话职责。**文件级别**，不要写函数级（太细）。

```markdown
| 文件 | 作用 |
|---|---|
| `backend/engine/foo.py` | ... |
| `backend/services/bar.py` | ... |
| `backend/models/baz.py` | ... |
```

## 7. 配置项汇总（按需）

只有 env 变量或 settings 字段时写。

```markdown
| 环境变量 | 默认 | 含义 |
|---|---|---|
| `XXX` | `value` | ... |

| Slot | 模型档位 | 用途 |
|---|---|---|
| `npc_agent` | 廉价 | NPC 对话 |
```

## 8. 数据库 schema（按需）

有专属表时写。**不复制完整 CREATE TABLE**——字段名 + 类型 + 一句话 + 关键 INDEX。完整 schema 在 `docs/data/schema.md`。

```sql
some_table (
  id, session_id, ...,
  related_npc,                -- 隔离锚点（0.A.3）
  embedding JSON,             -- 语义召回（1.B.2）
  created_at
)
INDEX (session_id, related_npc)
```

## 9. 测试覆盖（必写）

```markdown
| 测试文件 | 覆盖范围 |
|---|---|
| `tests/test_xxx.py` | ... |
```

改这个模块时知道动了哪个功能要跑哪些测试，比 `pytest tests/` 全跑省时间。

## 10. 已知短板与未来扩展（必写，跟矩阵 ❌🟡 联动）

按 P1 / P2 / P3 优先级分组，每条一句话：是什么、为什么没做、什么时机做。这一节直接对应矩阵里 `❌` 和 `🟡` 行——读者看完矩阵想"这个 gap 是怎么回事"就来这里查。

## 风格规范

### 写作风格

- **中文为主**。技术名词（asyncio.gather / SSE / cosine）保持英文
- 避免营销话术（"高效""强大""可扩展"）
- 避免空泛承诺（"未来会支持..."），用 "P2 / P3 待做" 标记并写明触发条件
- 不要复制代码片段超过 5 行——指出位置 + 一句话说明做什么

### 命名

- 文件名：英文小写 + 连字符（`world-creator.md` 不是 `worldCreator.md`）
- 章节标题中文 OK，但 `## 1. xxx` 编号必须保留
- 代码引用：`backend/engine/foo.py:123` 或 `function_name()`，不要写"foo.py 第 123 行"这类口语

### 准确性

- 写之前**必须读代码**。不能凭对项目的总体印象写
- 行号会随代码变化漂移 → 用关键标识符（`def process_action`）+ 大约行号（"约 line 297"）
- 引用 settings 字段或 SQL 列名时**逐字复制**，不要简写

### 同步策略

- 顶部"状态截至 YYYY-MM-DD"逐次维护
- 改代码且改动跨越多模块时，PR 必须连带改对应模块文档
- 出现"代码跟文档不一样"——以代码为准，立刻更新文档

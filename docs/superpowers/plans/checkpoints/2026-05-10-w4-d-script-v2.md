## Stage Flow Summary

| Stage | Phase key | Content |
|---|---|---|
| A | `research_pack` | `build_research_pack` with outline as description; inherits ip_canon from world_data |
| B | `script_base` | Single LLM: name / description / script_setting / script_type / difficulty / estimated_time |
| C | `events` | `build_events_data` with characters/locations/shared_events/lore_pack inherited from world_data |
| D | `endings` | Single LLM: 3-5 endings in v1 schema (name/description/condition/quality) |
| E | `playable` | Pure Python filter: is_image_target OR playable OR role_tag contains "主角" |
| F | `critic` | `validate_world_shape` + `light_critic_shared_events` (events IP consistency) + `moderate_world_payload` |

Result payload: `name, description, difficulty, estimated_time, script_setting, script_type, events, events_data, clues, endings, playable, research_pack, quality_warnings`

## Key Design Decisions

- **No world field regeneration**: `build_character_roster`, `build_characters_in_batches`, `build_lore_dimensions`, `build_lore_pack`, `build_shared_events` are never called from `create_script`
- **is_image_target mapping**: `_build_script_world_data` maps `wc.playable → is_image_target` since `WorldCharacter` has no `is_image_target` column; v2 agent treats both the same way
- **Feature flag**: `_run_script_generation` checks `settings.world_creator_v2_enabled` — same flag as world v2; v1 agent used otherwise
- **events_data v2**: Both `events` and `events_data` keys in result (compat + v2 new field); each entry is `EventDataEntry.model_dump()`

## Concerns / Notes

- `LorePack(**lore_pack_dict)` will fail if the world was published via v1 and `lore_pack` has a different shape — guarded with try/except fallback to `LorePack()`
- `_normalize_script_payload` in `admin.py` does not yet include `events_data` or `research_pack` — these are stored in `ScriptDraft.payload` but won't be preserved on draft save/reload. This is a pre-existing gap that can be addressed when the admin UI for script v2 is built.
- The feature flag `world_creator_v2_enabled` controls both world v2 and script v2 — if they need to diverge, a separate `script_creator_v2_enabled` flag should be added.

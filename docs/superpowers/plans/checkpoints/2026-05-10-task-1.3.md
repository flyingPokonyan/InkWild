### 1. `backend/schemas/research_pack.py`

New schema module defining three Pydantic models per spec §3.2:

```python
"""ResearchPack schema — 结构化研究产物，跨阶段保真容器。"""

PassageSource = Literal["tavily", "ip_probe", "admin_note"]

class Passage(BaseModel):
    """单个研究原文片段。"""
    id: str
    text: str
    tags: list[str] = Field(default_factory=list)
    source: PassageSource

class IPCanon(BaseModel):
    """LLM 自查 + Tavily 抽出的 IP 结构化知识。题材不需要时全字段为空数组。"""
    title_guesses: list[str] = Field(default_factory=list)
    canonical_names: list[str] = Field(default_factory=list)
    canonical_places: list[str] = Field(default_factory=list)
    iconic_objects: list[str] = Field(default_factory=list)
    lingo: list[str] = Field(default_factory=list)
    notable_events: list[str] = Field(default_factory=list)

class ResearchPack(BaseModel):
    """结构化研究产物容器，包含摘要、原文段、IP结构化知识。"""
    summary: str
    passages: list[Passage] = Field(default_factory=list)
    ip_canon: IPCanon = Field(default_factory=IPCanon)
```

**Key implementation details:**
- Uses Pydantic v2 with modern type hints (`str | None`, `list[str]`)
- `PassageSource` defined as `Literal["tavily", "ip_probe", "admin_note"]` to restrict valid sources
- All list fields default to empty via `Field(default_factory=list)`
- `ip_canon` field defaults to empty `IPCanon()` instance
- Follows project convention of simple models without validators (truncation logic deferred to builder layer)
- Module docstring included per spec

## Design Decisions

### 1. Schema Layer: No Truncation Logic

Per task spec ("truncation 由 builder 做，schema 层不强制"):
- `IPCanon.canonical_names` accepts unlimited items at schema layer
- Spec §3.5 specifies 200-item cap as enforced by `research_broker.py` builder, not schema validation
- Similar for passage count (100-item cap, 600-char per passage) — builder responsibility
- This keeps schema focused on structural validation, not business logic

### 2. Pydantic v2 Adoption

Project already uses Pydantic v2 extensively (confirmed in `schemas/case_board.py`, `schemas/generation_strategy.py`):
- Modern syntax: `str | None` instead of `Optional[str]`
- `Literal["tavily", "ip_probe", "admin_note"]` native support
- `Field(default_factory=list)` works identically in v1 and v2
- No breaking changes with existing schemas

### 3. Default Factory vs. Direct Default

Used `Field(default_factory=list)` and `Field(default_factory=IPCanon)` throughout:
- Prevents mutable default gotchas (shared list across instances)
- Matches project convention (see `generation_strategy.py`, `case_board.py`)
- Each new model gets its own empty list/IPCanon instance

### 4. Field Ordering

Ordered per spec §3.2:
- `Passage`: id, text, tags, source
- `IPCanon`: 6 knowledge dimensions in semantic grouping (titles → names → places → objects → lingo → events)
- `ResearchPack`: summary, passages, ip_canon

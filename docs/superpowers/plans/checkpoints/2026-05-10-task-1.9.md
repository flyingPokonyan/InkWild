## v1 Agent Attributes Adapted

The v1 `WorldCreatorAgent.__init__` stores:
- `self.llm` ŌåÉ `llm_router` (LLMRouter)
- `self.image_gen` ŌåÉ `image_generator` (ImageGenerator | None)
- `self.research_broker` ŌåÉ `ResearchBroker(tavily=..., web_searcher=..., synthesizer=...)`

When building `WorldCreatorAgentV2` from the v1 factory result:
```python
agent = WorldCreatorAgentV2(
    llm=base_agent.llm,
    image_gen=base_agent.image_gen,
    broker=base_agent.research_broker,   # ŌåÉ NOT .broker
)
```

**Key**: v1 uses `self.research_broker`, not `self.broker`. v2 internally stores it as `self.broker`.

## generation_feedback API (actual signatures)

```python
# phase + code as positional args; message derived from FEEDBACK_TEMPLATES; **meta for extras
progress_event(phase: str, code: str, **meta: object) -> dict
    # Returns: {"type": "progress", "phase": ..., "code": ..., "message": ..., "meta": meta}

warning_event(phase: str, code: str, **meta: object) -> dict
    # Returns: {"type": "warning", "phase": ..., "code": ..., "message": ..., "meta": meta}

result_event(data: dict, partial: bool = False, completed_phases: list[str] | None = None) -> dict
    # Returns: {"type": "result", **data}   ŌåÉ data is spread into the event

done_event() -> dict
    # Returns: {"type": "done"}

error_event(message: str, code: str = "generation_failed", phase: str = "general", **meta) -> dict
```

**Important**: Events use `"type"` key (not `"event"`). The `_record_event` method reads `event.get("type", "")` to determine the DB event_name.

## Design Decisions

1. **v2 constructor stores `self.broker`** (the ResearchBroker); the factory-bridge in `_run_world_generation` reads `base_agent.research_broker` to pass it in. This keeps the v2 class API clean without exposing the v1 naming.

2. **Test for v1 flag-disabled**: adapted to use a plain stub whose `create_world` is an async generator function directly (not a class method monkeypatch), because the stub_agent is a `MagicMock` not a real `WorldCreatorAgent` instance.

3. **`result_event` usage**: `result_event({"research_pack": pack_dict})` ŌĆö the dict is spread via `**data`, so the resulting event is `{"type": "result", "research_pack": {...}}`. The test checks `payload["research_pack"]` on the event dict directly.

4. **`warning_event` for not-yet-implemented phases**: uses `message=` as a kwarg in `**meta`, which surfaces in `meta.get("message", "")` fallback since the template for `(phase, "not_yet_implemented")` is not in FEEDBACK_TEMPLATES. This is correct ŌĆö the message is passed through via meta.

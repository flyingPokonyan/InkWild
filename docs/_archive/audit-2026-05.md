## 综合评级：B+

**最大优点**：治理意图清晰，9 档字号 + ESLint 锁死 + PR 自检表 + Storybook 是这套规范里最坚硬的部分，骨架配得上 Letterboxd-ish 的对标坐标。装饰克制（accent ≤ 1 处、绿色严禁装饰、字重禁 700+、卡片字段 ≤ 5）这几条是非常成熟的设计系统直觉。

**最大问题**：v2.2 把叙事正文从 serif 改 sans（visual §1.1 + play-mode §4.1）—— 这一刀直接削掉了产品向 Apple TV+/Letterboxd 看齐的最强承载点。"中文长段 sans 更舒适" 是工程舒适度，不是产品调性判断。Apple Books / Kindle 中文都用思源宋体而不是苹方，原因正在这里。**叙事是 play 页用户停留 90% 的内容，叙事字体决定的是产品定位**。

**整体定位**：一份执行得很好但在最关键审美决策上向"工具感"回退的规范文档。修掉 P0，整体上 A-；不修，长期看产品视觉气质会和文档对标坐标背离。

## P0 修订（影响产品调性，立即改）

### 1. 叙事正文字体从 serif 改 sans 是产品定位降级 ✅ 已决策（2026-05-09：保留 sans）
**位置**：`visual-principles.md` §1.1 表格 + v2.1→v2.2 摘要；`play-mode-spec.md` §4.1 + §8 决策记录 1
**决策结论**：经 `/dev/play-typeface-demo` 双栏 demo 桌面 + 手机对比验证，**保留 v2.2 sans**。决策记录写入 `play-mode-spec.md` §8 第 5 条。下面的"问题/修订"分析保留作为论据档案，不再行动。
- **问题**：v2.2 把 play 页叙事段落（用户停留最久的内容）从 Cormorant Garamond + Noto Serif SC 改成 Inter + PingFang SC。理由"中文长段 sans 更舒适"是工程舒适度判断，不是产品调性判断。结果：play 页视觉气质从"Apple TV+ 副刊式叙事"滑向"Notion/Linear 工具感"。Apple Books / Kindle / 微信读书中文长文都用 serif 不是偶然。Inter 在 visual §1.1 自己也属于"AI 味敏感"字体范围，把它放到产品最核心的叙事位置是错位。
- **修订**：回滚到 v2.1。叙事正文 = serif（Cormorant + Noto Serif SC），保留 line-height 1.85 + 760 阅读宽 + letter-spacing 0.005em。把 sans 留给 UI 组件、按钮、表单、消息系统提示。
- **验证方案**：用 frontend-design 出 `/dev/play-typeface-demo` 路由：同一段 800 字中文叙事正文，左右分栏对比 serif vs sans，760 宽 + 1.85 行高 + 暖金 italic 高亮短语 + 系统提示 caps。让 5 个用户在 30 秒内说"哪个像高品味叙事产品 / 哪个像聊天 AI"。
- **信心度**：🟡 主观（但是高置信度的主观——产品定位级别的判断）

### 2. frontend-spec §2.6 旧 alias 数值与新 token 实际值不一致 ✅ 已修（2026-05-09：删表）
**位置**：`frontend-spec.md` §2.6 旧 alias 表
**修订结论**：grep 显示老 alias 实际消费 0 处（仅 globals.css 自身定义），整张表删除，留一句话说明 globals.css 兜底定义下次清理 PR 一并删。下面分析保留作论据档案。
- **问题**：表中写 `--lv-h1 → --lv-t-display (64)`、`--lv-h2 → --lv-t-h1 (32)`、`--lv-h3 → --lv-t-h2 (20)`。但 §2.2 明确 `--lv-t-display` clamp 到 **112**（桌面）、`--lv-t-h1` 到 **48**、`--lv-t-h2` 到 **28**。alias 表数值是 v2.1 之前的旧值。任何按这张表迁移旧代码的工程师都会改错。
- **修订**：删掉数值括号，或改成 `→ --lv-t-display（48 / 112，clamp）`。或更狠：alias 表整张删掉（§10 已说"阶段 5 完成后整组删除"，何不现在就让它消失）。
- **验证方案**：`grep -r "lv-h1\|lv-h2\|lv-h3\|lv-lead" frontend/` 看到底有多少代码还在消费 alias。<10 处就直接删表 + 改完代码；>10 处就先改数值再排迁移。
- **信心度**：✅ 客观（数值错误可验证）

### 3. patch 长期游离破坏单一真理来源 ✅ 已修（2026-05-09：3 处 patch 全部归并主体）

### 4. cover-art-spec §5.2 色调约束没有验收路径 ✅ 已决策（2026-05-09：降级为 advisory）
**位置**：`cover-art-spec.md` §5.2 调色（橱窗一致性）
**决策结论**：现状无 LUT、无设计师签字流程，§5.2 整节降级为 advisory（不阻断 publish）。强制项收窄为 4 项可机检指标（黑场 / 白场 / 色温 / sRGB），其他 4 项作为出图人自检参考。明文写出"升级为强制的触发条件"（LUT 交付 或 设计负责人到岗）。§9 验收清单同步收紧措辞。
- **问题**：表中"高光偏暖：色相向 #c9b48a 靠拢 5%-8%"、"阴影偏冷：色相向 #1a2028 靠拢 3%-5%"、"胶片颗粒 grain 0.3-0.5（主观上'不平滑但不脏'）"——这些规则在 PS / Lightroom / Capture One 里**没有对应操作**。"色相靠拢 X%" 不是色彩工具的术语；"grain 0.3-0.5" 没说是 Lightroom Grain 滑块还是 ACR Texture 还是别的。结果：审稿人和出图人对同一张图能给完全不同的判断，§9 验收清单第 7 条形同虚设。LUT 文件交付前这是治理黑洞。
- **修订**：要么 (a) 立即把 LUT 文件 `taled-cinematic.cube` 做出来，§5.2 改为"过 LUT 后比对参考图，色差 ΔE < 5"；要么 (b) 把数值改成可操作的 Lightroom 滑块值（"色温 -5 到 -10、色调 +3 到 +8、饱和度 -10 到 -20、对比 +5 到 +15"）。
- **验证方案**：找 1 张已上线封面，让两个不同的人按 §5.2 现状打分，看打分是否一致。不一致 = 规则不可执行。
- **信心度**：✅ 客观（验收路径缺失是事实）

### 5. lv-pad-x 在两份文档说不同的话，并违反间距 9 档 ✅ 已修（2026-05-09：统一 48 / 16）
**位置**：`frontend-spec.md` §6 vs `visual-principles.md` §4 末尾备注
**修订结论**：globals.css 移动端 `--lv-pad-x` 由 20px → 16px (s-4)，对齐间距 9 档。frontend-spec §6 与 visual-principles §4 末尾备注同步改写。三处现在一致。
- **问题**：frontend-spec §6 写 `--lv-pad-x: 48px (移动端 20px)`；visual-principles §4 备注说 "globals.css 的 `--lv-pad-x: clamp(28px, 4vw, 56px)` 是无级缩放，桌面端硬切到 48 即可"。三处不一致：(a) frontend-spec 没说 clamp 的事，写的是 fixed；(b) visual-principles 说桌面 48 但移动端没给值；(c) frontend-spec 移动端写的 **20px** 是 §4 自己禁止的"破例值"之一。
- **修订**：统一到 `--lv-pad-x: clamp(16px, 4vw, 48px)` 或固定 `desktop 48 / mobile 16`（s-4），把 20 这个破例值消除。两份文档同步改。
- **验证方案**：grep `lv-pad-x` 看实际 globals.css 的真值，以代码为准更新两份 spec。
- **信心度**：✅ 客观（跨文档矛盾 + 违反自身规则）

## P2 修订（优化但不紧急）

1. **t-h3 clamp 跨度只有 16→18px**（frontend §2.2）—— clamp 化几乎是常量，移动桌面差 2px 几乎感觉不到，可以改 fixed 17px 简化。

## 不改的好决策（值得保留 / 表扬）

1. **卡片字段 ≤ 5（visual §7.1）**—— 这是整份文档最像 Letterboxd 的一条。Steam / itch.io / AI Dungeon 的卡片都是 8-12 字段堆砌；坚持 5 个字段是产品调性的硬骨头，**不要被产品同事"加个收藏数吧"说服**。
2. **装饰 accent 一屏 ≤ 1 处 + 绿色严禁装饰（visual §2.1-§2.2）**—— 极成熟的设计系统直觉。多数团队做不到，因为写需求的人总想"再多用一点品牌色"。
3. **字重禁 700+（visual §1.6）**—— 中文 sans 加粗在 dark 底易糊是真观察，不是教条。Apple 中文设计也避重粗。
4. **不做 IM 气泡 / 不做 typewriter（play-mode §4.2 §4.3）**—— 选了"小说排版"而不是"聊天 UI"作为底层模型，这个判断和 Letterboxd/Apple TV+ 路线一致。typewriter 性能差是次要原因，"分散注意力" 才是主要原因，这条洞察很值钱。
5. **ESLint 锁死 + CI 失败 = PR 拒绝（visual §14.5）**—— 把"禁止 inline text-[Xrem]"从口头规则做成 lint 规则是治理成熟度的分水岭。多数团队止步于"自检表打勾"，你们做到了机检。
